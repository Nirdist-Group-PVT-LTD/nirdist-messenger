Revised schema — PostgreSQL (migrated from MySQL)
Key MySQL → PostgreSQL changes
• BIGINT UNSIGNED AUTO_INCREMENT → BIGSERIAL or BIGINT GENERATED ALWAYS AS IDENTITY
• TINYINT → SMALLINT or BOOLEAN
• ENUM(...) → PostgreSQL ENUM type or VARCHAR + CHECK constraint
• TEXT for JSON arrays → JSONB (indexable, queryable)
• Media/files stay as TEXT paths or URLs, not database blobs; keep the storage layer abstract for Render disk now and S3 later
• Add phone_number, firebase_uid, and phone_verified_at for Firebase OTP verification
• Remove SET FOREIGN_KEY_CHECKS = 0 — not needed in PostgreSQL
• Story/Note expiry → handle with pg_cron or app scheduler (Spring @Scheduled)

WebRTC call flow
1. Caller sends offer via WebSocket (STOMP)
2. Signaling server relays to callee
3. ICE candidates exchanged
4. STUN: stun.cloudflare.com
5. TURN fallback: coturn on Render or Cloudflare TURN
6. Peer-to-peer media established
7. Call events logged to DB
Real-time messaging flow
1. Flutter connects via STOMP over WebSocket
2. JWT verified on handshake
3. Subscribe to /topic/room.{id}
4. Send to /app/chat.send
5. Spring broadcasts to room topic
6. Persisted to chat_message table
7. Push notification via FCM
JWT auth flow
1. Login → Spring Security validates
2. Returns Access Token (15 min)
3. Returns Refresh Token (7 days)
4. Flutter stores in flutter_secure_storage
5. Interceptor auto-refreshes on 401
6. Logout → revoke refresh token (DB)
Media upload flow
1. App requests an upload session from the API
2. App uploads media to the Spring Boot media endpoint
3. API writes the file to Render persistent disk and stores metadata in PostgreSQL
4. Stories expire via @Scheduled cleanup
5. Storage adapter stays abstract for a future S3-compatible swap
6. API serves media through authorized download URLs

Future-proofing checklist
✓ API versioning at /v1/ — add /v2/ without breaking old clients
✓ Flutter flavors: dev, staging, prod — different API base URLs
✓ Feature flags table in DB — toggle features without deploy
✓ Database migrations via Flyway (V1__init.sql, V2__add_follow.sql…)
✓ Storage adapter abstraction: local/Render disk now, S3-compatible later without client changes
✓ Firebase OTP flow: client verifies phone with Firebase, backend only trusts Firebase ID tokens
✓ Sentry for error tracking (Flutter + Spring Boot SDKs)
✓ OpenAPI / Swagger at /api-docs for team reference
✓ GitHub Actions CI: lint → test → Docker build → push to Render
