# Render Deployment

This backend is deployed from the monorepo root with the Render Blueprint at `render.yaml` and the Dockerfile in `back-end/Dockerfile`.

## Service Layout

- `render.yaml` defines one web service and one Render Postgres instance.
- `back-end/Dockerfile` builds the Spring Boot jar and runs it on the Render `PORT`.
- `back-end/src/main/java/com/nirdist/controller/HealthController.java` exposes `/api/health` for Render health checks.

## Environment Variables

| Name | Source | Purpose |
| --- | --- | --- |
| `JDBC_DATABASE_URL` | Render/Neon/Postgres connection string | Accepts either a raw `postgresql://...` URL or a full `jdbc:postgresql://...` URL. Raw URLs are normalized automatically at startup. |
| `DB_USER` | Render Postgres `user` | Database username. |
| `DB_PASSWORD` | Render Postgres `password` | Database password. |
| `JWT_SECRET` | Generated by Render | HS512 signing key for backend JWTs. |
| `JWT_EXPIRATION` | Optional override | JWT lifetime in milliseconds. Defaults to one day. |
| `FIREBASE_SERVICE_ACCOUNT_JSON` | Manual secret value | Firebase Admin service account JSON used only to verify phone OTP tokens. Use a Render secret env var. |
| `FIREBASE_CREDENTIALS_PATH` | Optional manual value | Path to a mounted Firebase credentials file, if you prefer file-based credentials over inline JSON for phone OTP verification only. |
| `FIREBASE_PROJECT_ID` | Optional manual value | Overrides the Firebase project id for the phone OTP verification boundary when needed. |
| `PORT` | Render-provided | HTTP port for the container. |

## Notes

- Flyway runs automatically at startup, so the Render Postgres instance is migrated as soon as the service boots.
- If you use `FIREBASE_CREDENTIALS_PATH`, you also need to provide the credential file yourself; the default Render setup is to store the JSON in `FIREBASE_SERVICE_ACCOUNT_JSON`.
- Firebase is not used for chat, media, or profile features here; it is only the phone OTP verification boundary before JWT issuance.
- The backend health check is `/api/health`.

## Local Docker Build

```bash
docker build -t nirdist-backend back-end
docker run --rm -p 8080:8080 \
  -e JDBC_DATABASE_URL=postgresql://user:password@host:5432/nirdist?sslmode=require \
  nirdist-backend
```

## Neon Notes

- Neon connection strings work through `JDBC_DATABASE_URL`.
- If the connection string already contains the username and password, the backend extracts them automatically.
- You can also provide `DB_USER` and `DB_PASSWORD` separately when you do not want credentials embedded in the URL.
- If a URL includes `channel_binding=require`, the backend replaces it with JDBC-compatible `channelBinding=disable`.
- If a Neon pooled hostname (`-pooler.`) is provided, the backend normalizer rewrites it to the direct Neon host automatically.
